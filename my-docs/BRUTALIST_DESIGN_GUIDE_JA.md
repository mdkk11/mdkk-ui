# BRUTALIST DESIGN GUIDE (JA)

この文書は、`mdkk-ui` における Brutalist デザインを、AI と人間の両方に同じ解像度で伝えるためのガイドです。  
目的は「無骨な見た目を真似ること」ではなく、「情報構造がすぐ読めて、判断が速いUI」を再現可能にすることです。

---

## 1. このガイドでいう Brutalist とは何か

### 1-1. 一言でいうと

Brutalist とは、装飾で雰囲気を作るのではなく、**構造・境界・文字・コントラストで意味をはっきり見せる**デザインです。

### 1-2. キーワード

- 構造が見える
- 強い階層
- 迷わない操作
- 曖昧さを減らす
- 無骨だが雑ではない

### 1-3. このデザインの目的

- ユーザーが重要情報を一瞬で見分けられる
- 操作できるものと、読むだけのものをすぐ区別できる
- 状態変化が明確で、迷いなく次の行動に移れる
- 画面ごとに雰囲気がぶれず、同じ設計思想を保てる

### 1-4. このデザインが目指していないもの

- ただ太い線を増やすこと
- ただ全部を黒く重くすること
- ただレトロっぽく見せること
- 可読性やアクセシビリティを犠牲にすること

### 1-5. よくある誤解

- 「Brutalist = 攻撃的で派手な見た目」
  実際は、派手さよりも情報構造の明快さが中心です。
- 「Brutalist = 色数を減らせば成立する」
  実際は、色数よりも意味の割り当てが重要です。
- 「Brutalist = 何でも四角くして太線にする」
  実際は、線の太さや形は階層を示すために使い分けます。

## 2. Visual Theme & Atmosphere

### 2-1. 画面の空気感

- クリーンだが緊張感がある
- 編集ツールや作業環境のように、道具感がある
- 装飾的というより、構造的
- 情報が整理されていて、押す場所と読む場所が明快

### 2-2. 画面全体の印象

- 背景はフラットではなく、必要に応じてグリッドや区画線で秩序を見せる
- セクションは「余白」だけでなく、「境界」と「面」で分離する
- 強調は少数に限定し、画面全体を叫ばせない

---

## 3. Color Roles

色は雰囲気のためではなく、**意味のために使います**。

### 3-1. 役割ごとの意味

| Role | 意味 | Light | Dark |
|------|------|-------|------|
| Canvas | 画面全体の基調面 | `#f5f3ee` | `#171717` |
| Panel | パネルやカードの面 | `#ffffff` | `#0a0a0a` |
| Ink | 主文字・主境界 | `#000000` | `#ffffff` |
| Emphasis | 強調、CTA、ハイライト | `#ffe500` | `#ffe500` |
| Destructive | 削除、危険操作 | `#ff0000` | `#ff0000` |

### 3-2. semantic の基本ルール

- `primary`: 主操作。light では黒、dark では白
- `accent`: 強調。黄を使い、CTA やハイライトに限定
- `destructive`: 危険操作。赤を使い、accent と意味を混ぜない
- `muted`: 補助情報や弱い面。主役にはしない

### 3-3. 色の使い方

- 同じ意味に複数の色を割り当てない
- 強調色は少数に絞る
- 赤と黄を同じ「重要そうな色」として併用しない
- 文字色、背景色、境界色のコントラストを保つ

---

## 4. Typography Rules

文字は装飾ではなく、**役割を伝えるUI部品**として扱います。

### 4-1. 使用フォント

- `sans`: `Inter`, system-ui, sans-serif
- `heading`: `Bricolage Grotesque`, `Inter`, sans-serif
- `serif`: `Playfair Display`, Georgia, serif
- `mono`: `Space Mono`, monospace

### 4-2. 役割ごとの使い分け

- `heading`: セクション見出し、強いタイトル
- `sans`: 本文、説明、通常ラベル
- `mono`: 補助ラベル、識別子、小さなタグ
- `serif`: 常用しない。特別な演出が必要な場合だけ

### 4-3. タイポグラフィの原則

- 見出し、ラベル、本文の役割を混ぜない
- 見出しは大きさだけでなく、ウェイトと字間で差を出す
- ラベルは uppercase と高めの字間で機能ラベルとして見せる
- 本文は読みやすさを優先し、過剰な強調を避ける

### 4-4. mdkk-ui の既定値

- `--brutal-track-label`: `0.06em`
- `--brutal-track-heading`: `0.08em`
- `--brutal-track-kicker`: `0.12em`

使い分けの目安:

- label: 小さな機能ラベル
- heading: 見出し
- kicker: さらに強い導入ラベルやセクションの先頭

---

## 5. Layout Principles

レイアウトでは、余白より先に**構造**を決めます。

### 5-1. 基本原則

- まず主従関係を決める
- 次に区画を分ける
- 最後に装飾を足す

### 5-2. レイアウトで守ること

- セクションは背景差や境界線で明確に分ける
- 主要操作は視線導線上にまとめる
- 関係のある要素は近く、関係のない要素は離す
- 余白は雰囲気づくりではなく、関係性の表示として使う

### 5-3. グリッドの扱い

- グリッドは装飾ではなく、秩序の可視化として使う
- 常時強く見せるのではなく、背景として薄く支える
- `mdkk-ui` の既定値は `32px` グリッド

---

## 6. Surface, Border, Radius, Shadow

Brutalist では「面」と「境界」が特に重要です。

### 6-1. 面の考え方

- Canvas は全体の土台
- Panel は独立した情報のかたまり
- 面が違うなら、境界か背景差のどちらかを必ず入れる

### 6-2. 境界線の使い分け

| Token | 値 | 用途 |
|------|----|------|
| `--brutal-border-subtle` | `2px` | 小さなコントロール |
| `--brutal-border-default` | `3px` | カード、ダイアログ、ボタン |
| `--brutal-border-strong` | `4px` | 区画分離、レール、トップバー |

### 6-3. Radius の原則

- 基本は `rounded-none`
- 丸みは原則として付けない
- 形そのものに意味がある場合だけ例外を認める

### 6-4. Shadow の原則

- 影は立体感の演出ではなく、把握を助けるために使う
- ブラーよりも「オフセット」を重視する
- `mdkk-ui` では `2px / 4px / 8px` の offset shadow を使う

---

## 7. State & Interaction

状態変化は、色だけでなく**境界・濃度・文字・反転**でも伝えます。

### 7-1. 必須の状態

- default
- hover
- active / pressed
- disabled
- error / invalid
- pending / loading

### 7-2. 守るべきこと

- hover で少し変わる
- pressed で明確に反応する
- disabled は押せないことがすぐ分かる
- destructive は通常操作から視覚的に分離する
- focus は必ず見えるようにする

### 7-3. 禁止

- 色だけで状態差を出す
- disabled なのに hover と見分けがつかない
- destructive が accent と同じテンションで見える

---

## 8. Do / Don't

### Do

- 構造が先、装飾は後
- 強調は少数に絞る
- 色・境界・文字の意味を固定する
- 見た瞬間に優先順位が読める状態を目指す

### Don't

- 全部を強調して階層を壊す
- 画面ごとに色の意味を変える
- Brutalist を「荒っぽい見た目の模倣」にする
- 可読性を下げてでも雰囲気を優先する

### レビュー時の確認項目

1. 重要情報の優先順位が一瞬で分かるか
2. 操作可能要素が、読むだけの要素と混ざっていないか
3. 状態差分が色以外でも判別できるか
4. accent と destructive の意味が混線していないか
5. 装飾を減らしても構造が残るか

## 9. mdkk-ui Quick Reference

`mdkk-ui` に落とし込むときの最小マッピングです。

### 9-1. 主要 `--brutal-*` 変数

- `--brutal-canvas`: 画面全体の基調面
- `--brutal-panel`: パネル面
- `--brutal-ink`: 主文字・主境界
- `--brutal-ink-inverse`: 反転文字
- `--brutal-emphasis`: 強調色
- `--brutal-grid-size`: グリッド間隔
- `--brutal-grid-line`: グリッド線色

### 9-2. 主要 `brutal-*` ユーティリティ

- `brutal-canvas`: グリッド背景つきの基調面
- `brutal-topbar`: 強い区切りを持つトップバー
- `brutal-rail`: サイドレール面
- `brutal-label`: ラベル用タイポ
- `brutal-kicker`: 導入ラベル
- `brutal-heading`: 見出し用タイポ
- `brutal-chip`: 小さな強調ラベル
- `brutal-chip-inverse`: 反転ラベル
- `brutal-button`: Brutalist文脈のボタン基底

### 9-3. 運用メモ

- 先に抽象原則を決めてから token に落とす
- まず既存 semantic を使い、新しい色意味は安易に増やさない
- 値を変える前に、意味が変わっていないかを確認する

## 10. AI に渡すときの最小テンプレ

以下をそのまま AI に渡し、`[]` の中だけ埋めてください。

```md
あなたは Brutalist デザイン思想に従って UI を設計するアシスタントです。
この文脈での Brutalist は、無骨な見た目の模倣ではなく、情報構造を明快にし、操作判断を速くする設計です。

[画面の目的]
- [この画面でユーザーに達成してほしいこと]

[対象ユーザー]
- [例: 初回ユーザー / 管理者 / 編集担当]

[必ず守ること]
- 装飾より構造を優先する
- 重要度はレイアウト、文字、境界、色で表現する
- accent と destructive の意味を混ぜない
- 状態差分を色だけに依存しない
- 可読性とアクセシビリティを下げない

[出力してほしい内容]
1. 情報の優先順位
2. レイアウト方針
3. 色の役割分担
4. タイポの役割分担
5. 状態表現ルール
6. Do / Don't
```

### AI に追加で指定すると精度が上がる項目

- PC とモバイルでどこまで構造を保ちたいか
- どこを一番強く見せたいか
- destructive 操作があるか
- 入力フォーム中心か、閲覧中心か

- モーションのルール
- 画面種別ごとの優先パターン
