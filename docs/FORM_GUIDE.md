## 1. 設計方針

### 1.1 日本ユーザー特有の配慮事項

英語圏のUXガイドラインをそのまま適用することは日本市場では失敗の原因になります。以下の点を必ず考慮してください。

| 観点 | 日本での対応方針 |
|------|----------------|
| **敬語・丁寧語** | エラーメッセージや案内文は敬体（〜です・〜ください）で統一し、命令形は避ける |
| **全角 / 半角の混在** | 日本語入力では全角・半角が混在するため、フォーム側で自動変換（正規化）する |
| **姓・名の分割** | 法的書類・業務システムでは「姓」「名」を別フィールドにするのが実務標準 |
| **郵便番号からの自動補完** | 〒XXX-XXXX 入力 → 都道府県・市区町村を自動入力するのがUX標準（国内サービスのみ） |
| **携帯電話番号フォーマット** | 090/080/070-XXXX-XXXX の3ブロック表示を期待するユーザーが多い |

### 1.2 基本設計原則

- **必要最小限の項目設計：** 個人情報保護の観点からも不要な項目の収集はリスクになる（個人情報保護法）
- **明示的な同意取得：** 個人情報の利用目的はフォーム送信前に明示し、同意チェックボックスを設置する（必須）
- **ガイド・プレフィル活用：** 入力例・ヒント・自動補完でユーザーの認知負荷を下げる
- **重複入力の排除：** メール再確認入力は廃止し、確認メール＋コピペ許容で代替する

## 2. フィールド構造・ラベル設計

### 2.1 ラベル設計の原則

- **全ての入力に常時表示ラベルを付ける：** `placeholder` をラベル代用にすると入力中に消えてアクセシビリティが崩壊する
- **HTMLの紐付けは必須：** `<label for="id">` と `input` の `id` を一致させる（スクリーンリーダー対応）
- **ラベル位置は入力欄の上部：** 左配置は狭いモバイル画面で折り返しが起きるため、上部配置を推奨

### 2.2 必須・任意の表示

| ✅ 推奨（日本標準） | ❌ 避けるべき設計 |
|-------------------|----------------|
| 必須項目には「必須」バッジを付ける（赤背景・白文字） | `*` マークのみで必須を示す（意味が伝わらない場合あり） |
| 任意項目は「任意」バッジで明示する（グレー） | 必須だけ示して任意を暗黙にする（何が省略可能か不明） |
| ラベル横にバッジを配置しCSSで視認性を確保する | ページ冒頭に「*は必須」の凡例だけ書いてマークで済ませる |

### 2.3 プレースホルダの正しい使い方

> ⚠️ **重要**  
> プレースホルダはヒント（入力例）用途に限定してください。ラベルの代用に使うと、入力中に消えてユーザーが何を入力すべきか忘れます。またスクリーンリーダーは `placeholder` を読み上げない実装が多く、アクセシビリティ上も問題です。

- **良い例：** 氏名フィールドに `placeholder="例：山田 太郎"`
- **悪い例：** `placeholder="お名前"` でラベルを省略する

## 3. 主要フィールド別 実装標準

### 3.1 氏名フィールド

英語圏では「フルネーム1フィールド」が推奨されますが、日本では用途によって分けます。

| ユースケース | 推奨フィールド構成 |
|-------------|-----------------|
| **BtoC 一般サービス** | 姓名を1フィールドにまとめる（入力ストレス低減） |
| **BtoB・業務システム** | 「姓」「名」を別フィールドに分割する（ソート・帳票処理のため） |
| **公的サービス・医療・金融** | 「姓（漢字）」「名（漢字）」「姓（カナ）」「名（カナ）」の4フィールド推奨 |
| **カナ入力** | ふりがなは全角カタカナに自動変換する（IME入力の平仮名を正規化） |

---

### 3.2 住所フィールド

#### 基本方針

**住所入力欄は分割しない。** 都道府県・市区町村・町名・番地・建物名をすべて別フィールドにすることは、入力の手間と認知負荷を増やします。郵便番号検索で大部分を自動補完し、残りを1つのテキスト欄に入力させる構成が現在の標準です。

#### 国内向けサービス：郵便番号検索による入力簡素化

> 💡 **ベストプラクティス**  
> 郵便番号（7桁）入力 → zipcloud 等の無料APIで都道府県・市区町村・町名を自動補完 → 番地・建物名のみ手入力。これにより入力ミスを劇的に減らせます。

**郵便番号フィールドの実装詳細**

- **フォーマット：** ハイフンあり（`123-4567`）とハイフンなし（`1234567`）の両方を受け付ける
  - 入力値からハイフン・スペース・全角数字を除去し、7桁数字として正規化してからAPIに渡す
  - 表示上のハイフンは入力中に自動挿入しても構わないが、しなくても許容する
- **全角数字の自動変換：** `１２３４５６７` → `1234567` に変換する（日本語キーボードからの誤入力対策）
- **API検索タイミング：** 7桁確定時点で自動実行（送信ボタン不要）
- **補完後の編集：** 自動補完された都道府県・市区町村・町名は編集可能にする
- **推奨ライブラリ：** [zipcloud](https://zipcloud.ibsnet.co.jp/)（無料・REST API・CORS対応）

**住所フォームの推奨構成（国内向け）**

```
〒 [___-____]  ← 郵便番号（ハイフンあり/なし両対応、7桁で自動検索）
都道府県 [セレクト or テキスト] ← 自動補完・編集可
市区町村・町名 [テキスト]       ← 自動補完・編集可
番地・建物名 [テキスト]         ← 手入力（全角・半角混在を許容）
```

**属性設定例**

```html
<input
  type="text"
  id="postal-code"
  name="postal_code"
  inputmode="numeric"
  maxlength="8"
  autocomplete="postal-code"
  placeholder="例：123-4567"
  aria-label="郵便番号"
>
```

#### 国際向けサービス：郵便番号検索は外すことを検討

国際サービスでは郵便番号の書式・用途が国によって大きく異なります。

- 国によっては郵便番号が存在しない、または任意項目（アイルランド・香港など）
- 国ごとの自動補完APIを全対応するのはコスト面で非現実的
- 推奨構成：「国」セレクトボックスで国内/海外を切り替え、国内選択時のみ郵便番号検索を有効化する
- 海外住所は「Address Line 1 / Address Line 2 / City / State / ZIP / Country」の国際標準レイアウトに切り替える

#### autocomplete 属性とブラウザ自動入力の注意点

> ⚠️ **クロスブラウザ対応の限界**  
> ブラウザ自動入力（autocomplete）はSafariとChromeで仕様の解釈が異なり、完全な一致は困難です。特に住所フィールドの自動入力挙動はブラウザ・OS・バージョンによって差異があり、意図しないフィールドに値が入るケースも報告されています。autocomplete属性は設定すべきですが、「完璧に動く」と期待せず、手入力での編集を前提としたUX設計をしてください。

| フィールド | `autocomplete` 属性値 |
|-----------|----------------------|
| 郵便番号 | `postal-code` |
| 都道府県 | `address-level1` |
| 市区町村 | `address-level2` |
| 町名・番地 | `address-line1` |
| 建物名・部屋番号 | `address-line2` |

---

### 3.3 電話番号

- **入力形式：** 1フィールドで受け付け、ハイフンは後処理で正規化（`090-1234-5678` も `09012345678` も許容）
- **自動変換：** 全角数字 → 半角数字に自動変換（日本語キーボードでの誤入力対策）
- **type属性：** `<input type="tel">` を使用してモバイルでテンキーを表示する
- **バリデーション：** 10〜11桁数字（ハイフン除外後）でゆるく検証。厳格すぎる正規表現は新番号体系に対応できない

### 3.4 メールアドレス

- 再入力確認フィールドは廃止する（確認メール＋コピペ許容で代替）
- `type="email"` を使用し、ブラウザの基本バリデーションを活用する
- キャリアメール（docomo/au/softbank）は特殊フィルタ設定があるため、確認メールの到達率に注意する
- `autocomplete="email"` を設定してパスワードマネージャー・オートフィルを支援する

### 3.5 マイナンバー（個人番号）

> 🔴 **法令対応必須**  
> マイナンバーの収集・保管・利用は「番号利用法（マイナンバー法）」で厳格に規制されています。収集が法的に認められている場合のみフォームに含め、保管・アクセス制御・廃棄についても法令に従ってください。

- 12桁数字のみ許容、マスク表示（入力中は `*` で隠す）を実装する
- 入力後の確認表示でも末尾4桁のみ表示する
- SSL/TLS通信を必須とし、サーバーログへの記録を禁止する
- 利用目的の明示と本人確認の証拠記録を保持する

## 4. パスワード設計

### 4.1 推奨設計

- **パスワード強度メーターを表示：** 弱・中・強のリアルタイム表示でユーザーを適切な強度に誘導
- **表示/非表示切替ボタン：** 入力ミス削減のため必須（NIST SP800-63B 推奨）
- **autocomplete対応：** 新規登録時は `autocomplete="new-password"`、ログイン時は `autocomplete="current-password"` を設定
- **最低文字数は12文字以上：** NISTの最新推奨（旧来の「8文字」は脆弱）
- **絵文字・全角文字も許容：** 文字種を制限しすぎると逆にエントロピーが下がる

### 4.2 廃止すべき旧来設計

| ✅ 推奨（現行標準） | ❌ 廃止すべき設計 |
|-------------------|----------------|
| 定期変更を強制しない（漏洩が確認された場合のみ変更を促す） | 90日ごとのパスワード強制変更（NIST 2017以降で否定されている） |
| パスワードをハッシュ化してDBに保存（bcrypt/Argon2推奨） | 秘密の質問（生年月日・ペットの名前等）による認証 |
| 漏洩済みパスワードリストと照合してブロック | SMS OTPのみの二段階認証（SIMスワップ攻撃に脆弱） |
| FIDO2/WebAuthnによるパスキー認証の提供 | 「英数字記号を組み合わせること」の強制ルール（複雑性ルールはNISTで否定） |

## 5. バリデーション・エラー設計

### 5.1 バリデーションの二重実装（必須）

> ⚠️ **重要原則**  
> クライアントサイドのバリデーションはUX向上のためのものです。セキュリティはサーバーサイドの検証のみで担保します。クライアント検証はJavaScriptを無効にしただけで回避できるため、サーバー検証を省略することは重大なセキュリティリスクです。

| レイヤー | 役割と実装内容 |
|---------|--------------|
| **クライアント側（UX目的）** | リアルタイム検証・即時フィードバック・送信前チェック |
| **サーバー側（セキュリティ目的）** | 全入力の再検証・SQLインジェクション対策・XSS対策・ビジネスロジック検証 |
| **検証のタイミング** | フォーカスアウト時に検証を開始（入力中の検証は煩わしい） |
| **検証の優しさ** | 電話・メール等はゆるい正規表現で検証（厳格すぎると正当な入力を弾く） |

### 5.2 日本語エラーメッセージの標準

エラーメッセージは「なぜ・どうすれば」を明確に伝える敬体の日本語で記述します。

| エラー種別 | 推奨エラーメッセージ（敬体） |
|-----------|--------------------------|
| **メールアドレス形式エラー** | 「メールアドレスの形式が正しくありません。「@」を含む形式（例：name@example.com）でご入力ください。」 |
| **必須入力未入力** | 「「お名前」は必須項目です。ご入力をお願いします。」 |
| **パスワード文字数不足** | 「パスワードは12文字以上でご設定ください。現在：X文字」 |
| **電話番号形式エラー** | 「電話番号は10〜11桁の数字でご入力ください（ハイフン不要）。」 |
| **郵便番号エラー** | 「郵便番号が見つかりませんでした。7桁の数字でご入力ください。」 |

### 5.3 エラー表示のアクセシビリティ

- **視覚：** 赤色のボーダー＋エラーアイコン＋テキストを組み合わせる（色覚障害者は色のみでは判別不可）
- **スクリーンリーダー：** `role="alert"` または `aria-live="assertive"` でエラー発生を即時通知する
- **aria-describedby：** エラーメッセージ要素のIDをinputの `aria-describedby` に設定して関連付ける
- **フォーカス移動：** 送信失敗時は最初のエラーフィールドにフォーカスを移動させる

## 6. モバイル・キーボード UX

### 6.1 input モード・type属性の使い分け（日本語対応）

| フィールド種別 | 推奨属性設定 |
|--------------|------------|
| **氏名（漢字）** | `type="text"` / `inputmode="text"` |
| **ふりがな（カタカナ）** | `type="text"` / `inputmode="text"` + IMEモード指示テキスト |
| **メールアドレス** | `type="email"` / `inputmode="email"` / `autocomplete="email"` |
| **電話番号** | `type="tel"` / `inputmode="numeric"` / `autocomplete="tel"` |
| **郵便番号** | `type="text"` / `inputmode="numeric"` / `maxlength="8"` / `autocomplete="postal-code"` |
| **数量・金額** | `type="text"` / `inputmode="numeric"` / `min`・`max` 指定（`type="number"` は挙動差異があり非推奨） |
| **パスワード（新規）** | `type="password"` / `autocomplete="new-password"` |
| **パスワード（ログイン）** | `type="password"` / `autocomplete="current-password"` |

### 6.2 タッチ・モバイル設計

- タップターゲットは最低44×44px（Apple HIG・Material Design共通基準）
- ラジオボタン・チェックボックスは `label` タグ全体をタップ領域にする
- フォームは縦一列レイアウトを基本とする（横並びは幅480px以上のみ）
- 日本語入力（IME）確定前にバリデーションを発火させない（`compositionstart` イベントを制御）
- 数字入力は `type="number"` でなく `type="text"` + `inputmode="numeric"` を推奨（ブラウザの挙動差異を回避）

### 6.3 キーボードアクセシビリティ

- 全フォーム要素はTabキーで操作可能にする（`tabindex="-1"` の乱用禁止）
- カスタムUI（日付ピッカー等）はキーボード操作（矢印キー・Enter・Escape）を実装する
- フォーカスリング（`focus-visible`）を削除しない（見た目の理由で消すことは不可）
- 論理的なタブ順序を維持する（CSSでレイアウト変更してもDOM順序と一致させる）

## 7. アクセシビリティ

> 📋 **法令背景**  
> 2024年（令和6年）4月施行の改正障害者差別解消法により、民間事業者にも合理的配慮の提供が義務化されました。WebサービスのアクセシビリティはJIS X 8341-3（WCAG 2.1 AA相当）への準拠が実質的な標準となっています。

### 7.1 フォームに関するWCAG 2.1 AA 必達要件

| WCAG 達成基準 | フォームでの実装内容 |
|-------------|------------------|
| **1.3.1 情報と関係性** | ラベル・グループ・必須を構造的に実装（`for`/`id`、`fieldset`/`legend`） |
| **1.3.5 入力目的の特定** | `autocomplete` 属性で入力目的を明示（氏名・住所・メール等） |
| **1.4.3 コントラスト比** | テキストは4.5:1以上、大文字テキストは3:1以上のコントラスト比 |
| **2.4.7 フォーカスの可視化** | キーボードフォーカスを視覚的に明示する（アウトラインを消さない） |
| **3.3.1 エラーの特定** | 入力エラーをテキストで識別・説明する |
| **3.3.2 ラベルまたは説明** | 入力に説明が必要な場合は明示的な説明文を提供する |
| **4.1.2 名前・役割・値** | 全UIコンポーネントにARIA属性で名前と役割を提供する |

### 7.2 ARIAの正しい使い方

- **`aria-required="true"`：** 必須フィールドのinputに付与（HTMLの `required` 属性と併用）
- **`aria-invalid="true"`：** バリデーション失敗時にinputに付与し、エラー状態をプログラム的に通知
- **`aria-describedby`：** ヒントテキスト・エラーメッセージのIDを設定してinputに関連付ける
- **`aria-live="assertive"`：** エラー表示領域に付与してスクリーンリーダーに即時読み上げさせる
- **`role="group"` + `aria-labelledby`：** ラジオボタングループの `fieldset` 代替として使用する

## 8. セキュリティ・法令対応

### 8.1 入出力の安全処理（3ステップ）

> 🛡️ **実装順序**  
> ① 入力時：バリデーション（形式・型・長さの検証）  
> ② 保存前：サニタイズ（危険な文字の除去・正規化）  
> ③ 出力時：エスケープ（XSS対策。HTMLコンテキストに応じたエスケープ）  
> この3つは別フェーズであり、混同しないこと。

| 脅威・リスク | 対策と実装方法 |
|-----------|--------------|
| **SQLインジェクション対策** | プリペアドステートメント（パラメータバインド）を必ず使用。文字列結合でSQLを構築することは絶対禁止 |
| **XSS対策** | 出力時にHTMLエスケープ（`<` `>` `&` `"` `'` を変換）。`innerHTML` への直接代入禁止 |
| **CSRF対策** | CSRFトークン（フォームに hidden フィールドとして埋め込み）を全POST/PUT/DELETEに実装 |
| **レート制限** | 同一IPからの連続送信を制限（フォームスパム・クレデンシャルスタッフィング対策） |
| **HTTPS必須** | フォームページと送信先エンドポイントは必ずHTTPS。HTTPSでないフォームはブラウザが警告を表示 |

### 8.2 日本の法令対応チェックリスト

| 法令・規制 | フォームでの対応事項 |
|-----------|------------------|
| **個人情報保護法** | 収集目的の明示（同意取得）／第三者提供の制限／保管期限の設定と廃棄／プライバシーポリシーへのリンク |
| **特定商取引法** | ECサイトの申込内容確認画面の実装必須／「申込む」ボタンのラベルを法令に沿って明確化／クーリングオフ告知（対象取引） |
| **電子契約法** | 電子署名・タイムスタンプによる完全性担保／エラー訂正手段の提供（消費者向け） |
| **改正電気通信事業法** | クッキー同意管理（オプトイン方式）／トラッキング情報の外部送信規律への対応 |
| **不正アクセス禁止法** | ブルートフォース対策（ロックアウト・CAPTCHA）／不審ログインの検知・通知 |

### 8.3 同意取得フォームの設計

- プライバシーポリシー・利用規約は全文を読める状態でリンクを設置する
- チェックボックスはデフォルトOFFで実装（デフォルトONはダークパターンとして問題）
- マーケティングメールの同意は個人情報収集の同意と分けて取得する
- 未成年者（18歳未満）向けサービスでは保護者の同意取得フローを実装する
- 同意の日時・バージョン・IPをログに記録する（法的証拠として機能）
